KIP-848 Implementation Coverage

Purpose

мқҙ л¬ём„ңлҠ” Redis Stream Coordinatorк°Җ KIP-848мқҳ м–ҙл–Ө м•„мқҙл””м–ҙлҘј кө¬нҳ„н•ҳкі , м–ҙл–Ө л¶Җл¶„мқ„ Redis Stream м ңм•Ҫм—җ л§һкІҢ л°”кҫёкұ°лӮҳ м ңмҷён–ҲлҠ”м§Җ м •лҰ¬н•ңлӢӨ. лӘ©н‘ңлҠ” Kafka protocol нҳёнҷҳмқҙ м•„лӢҲлқј KIP-848мқҳ coordinator-managed rebalance лӘЁлҚёмқ„ Redis Stream shard ownershipм—җ м Ғмҡ©н•ҳлҠ” кІғмқҙлӢӨ.

Implemented From KIP-848

  • Coordinator-driven rebalance: memberк°Җ ownerлҘј мөңмў… кІ°м •н•ҳм§Җ м•Ҡкі  coordinatorк°Җ target assignmentлҘј кі„мӮ°н•ңлӢӨ.
  • Declarative target assignment: coordinatorлҠ” groupмқҙ мҲҳл ҙн•ҙм•ј н•  desired shard ownershipмқ„ м ҖмһҘн•ңлӢӨ.
  • Current assignment reporting: memberлҠ” heartbeatл§ҲлӢӨ мӢӨм ң owned/revoking shard мғҒнғңлҘј ліҙкі н•ңлӢӨ.
  • Current assignment validation: coordinatorлҠ” target assignmentмҷҖ мқҙм „м—җ мҲҳлқҪн•ң current assignment кё°мӨҖмңјлЎң stale ownership reportлҘј fencingн•ңлӢӨ.
  • Heartbeat assignment channel: coordinatorлҠ” heartbeat responseлЎң assigned/pending shard assignmentмҷҖ fencing statusлҘј м „лӢ¬н•ңлӢӨ.
  • Incremental reconciliation: group-wide stop-the-world barrier м—Ҷмқҙ ліҖкІҪлҗң member/shardл§Ң revoke/assignн•ңлӢӨ.
  • Revoke-before-assign dependency: кё°мЎҙ ownerмқҳ revoke ack лҳҗлҠ” member expiration нҷ•мқё м „ мғҲ ownerм—җкІҢ assignн•ҳм§Җ м•ҠлҠ”лӢӨ.
  • Epoch model: groupEpoch, assignmentEpoch, memberEpochмңјлЎң group metadata, target assignment, member fencingмқ„ кө¬л¶„н•ңлӢӨ.
  • Coordinator event loop: heartbeat, metadata change, member lease expirationмқ„ coordinator loopм—җм„ң мҲңм°Ём ҒмңјлЎң нҸүк°Җн•ңлӢӨ.
  • Session timeout and fencing: member-lease-ttl лҸҷм•Ҳ heartbeatк°Җ м—ҶлҠ” memberлҘј fencingн•ҳкі  shardлҘј мһ¬н• лӢ№н•ңлӢӨ.
  • Sticky movement minimization: assignmentлҠ” sticky partition м „м ңлЎң кё°мЎҙ ownerлҘј мөңлҢҖн•ң мң м§Җн•ңлӢӨ.

Redis-Specific Adaptations

  • Kafka broker coordinator лҢҖмӢ  Coordinator API serverк°Җ group metadataмҷҖ assignmentлҘј кҙҖлҰ¬н•ңлӢӨ.
  • Kafka broker wire protocol лҢҖмӢ  custom HTTP Admin/Heartbeat/Monitoring APIлҘј мӮ¬мҡ©н•ңлӢӨ.
  • Kafka topic partition лҢҖмӢ  Redis Stream shard key {streamVersion, shardIndex}лҘј assignment лӢЁмң„лЎң мӮ¬мҡ©н•ңлӢӨ.
  • Kafka offset commit fencingмқҖ кө¬нҳ„н•ҳм§Җ м•ҠлҠ”лӢӨ. Redis Stream read/ack fencingмқҖ member data-planeмқҙ coordinatorмқҳ assignment/member epochмқ„ кё°мӨҖмңјлЎң м Ғмҡ©н•ңлӢӨ.
  • Kafka partition expansion лҢҖмӢ  stream version migrationмңјлЎң shard scale-out/inмқ„ мІҳлҰ¬н•ңлӢӨ.
  • Kafka ConsumerGroupHeartbeat RPCлҠ” internal coordinator API лҳҗлҠ” Redis mailbox request/responseлЎң кө¬нҳ„н•ңлӢӨ.
  • member idлҠ” member runtimeмқҙ мғқм„ұн•ҳкі  coordinatorк°Җ epoch/fencing мғҒнғңлҘј кҙҖлҰ¬н•ңлӢӨ.
  • Kafkaмқҳ group.consumer.heartbeat.interval.msмҷҖ group.consumer.session.timeout.msлҠ” coordinator heartbeat-intervalкіј member-lease-ttlлЎң лӢЁмҲңнҷ”н•ңлӢӨ.
  • KIP-848мқҳ server-side assignor м„ нғқмқҖ м ңкіөн•ҳм§Җ м•ҠлҠ”лӢӨ. sticky partition assignmentлҠ” м„Өкі„ м „м ңмқҙлӢӨ.
  • Basic Auth, Monitoring API, Redis м ‘мҶҚ YAML, Admin API default к°’мқҖ Redis coordinator мҡҙмҳҒмқ„ мң„н•ң м¶”к°Җ кі„мёөмқҙл©° KIP-848 protocol к°ңл…җмқҙ м•„лӢҲлӢӨ.

Not Implemented

  • Kafka wire protocol compatibility.
  • Kafka ConsumerGroupPrepareAssignment / ConsumerGroupInstallAssignment client-side assignment delegation.
  • assignor negotiation, assignor version probing, assignor metadata compatibility handling.
  • regex topic subscription and server-side topic metadata resolution.
  • Kafka static membership semantics such as temporary leave with instance id replacement.
  • Kafka offset commit/fetch APIs and topic-id based offset handling.
  • Kafka group dynamic config API. Redis Stream Coordinator uses its own Admin API for create, shard scale, consumer concurrency update, and rollback.

Coverage Notes

KIP-848 says the new protocol moves complexity from clients to the group coordinator, removes a global synchronization barrier, stores target/current assignment, and uses heartbeat to carry member state and assignment. This design implements those parts directly, then replaces Kafka-specific broker, partition, offset, and protocol concerns with Redis metadata keys, stream version migration, and member data-plane boundaries.
